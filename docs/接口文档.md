## 1. 通用约定

### 1.1 响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

### 1.2 分页请求参数

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| page | int | 否 | 1 | 页码 |
| size | int | 否 | 20 | 每页条数 |
| sort | string | 否 | createdAt,desc | 排序字段,方向 |

### 1.3 分页响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [],
    "total": 100,
    "page": 1,
    "size": 20,
    "pages": 5
  }
}
```

### 1.4 认证方式

除登录、注册、Token 回调等公开接口外，所有接口需在 Header 中携带：

```
Authorization: Bearer <token>
```

### 1.5 角色权限

| 角色 | 权限说明 |
|------|----------|
| ADMIN | 系统管理员，拥有全部权限 |
| EXPERT | 农技专家，可审核、制定防治方案 |
| MANAGER | 农场管理者，查看数据、报告 |
| VISITOR | 访客，只读权限 |

## 2. 认证模块 (Auth)

### 2.1 用户登录

**POST** `/auth/login`

**请求体：**

```json
{
  "username": "string",
  "password": "string",
  "rememberMe": false
}
```

**响应：**

```json
{
  "code": 200,
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "expiresIn": 86400,
    "userInfo": {
      "id": "uuid",
      "username": "zhangsan",
      "name": "张三",
      "role": "MANAGER",
      "phone": "13800138000",
      "email": "zhangsan@example.com",
      "avatar": "/images/avatar/zhangsan.jpg",
      "companyId": "company-uuid",
      "approved": true
    }
  }
}
```

**错误码：**

| code | 说明 |
|------|------|
| 40001 | 用户名或密码为空 |
| 40002 | 用户名或密码错误 |
| 40003 | 账号已被禁用 |

---

### 2.2 发送邮箱验证码

**POST** `/auth/send-otp`

**请求体：**

```json
{
  "email": "user@example.com",
  "type": "LOGIN"
}
```

**响应：**

```json
{
  "code": 200,
  "message": "验证码已发送，请查收邮箱"
}
```

**错误码：**

| code | 说明 |
|------|------|
| 40010 | 邮箱格式不正确 |
| 40011 | 验证码发送过于频繁，请1分钟后重试 |

---

### 2.3 邮箱验证码登录

**POST** `/auth/login-by-otp`

**请求体：**

```json
{
  "email": "user@example.com",
  "otp": "123456"
}
```

**响应：** 同 2.1 登录响应

---

### 2.4 退出登录

**POST** `/auth/logout`

**请求头：** `Authorization: Bearer <token>`

**响应：**

```json
{
  "code": 200,
  "message": "退出成功"
}
```

## 4. 企业/租户模块 (Company)

> **多租户说明：** 系统采用基于 `company_id` 的多租户隔离架构。每个企业拥有独立的数据空间，所有业务表均包含 `company_id` 字段。用户注册后需通过邀请码加入企业才能访问业务功能。

### 4.1 验证邀请码

**POST** `/company/validate-invite`

**说明：** 公开接口，用于验证邀请码是否有效并获取企业名称。

**请求体：**

```json
{
  "inviteCode": "TF2026"
}
```

**响应：**

```json
{
  "code": 200,
  "data": {
    "valid": true,
    "companyName": "TreeForge"
  }
}
```

**错误码：**

| code | 说明 |
|------|------|
| 40090 | 邀请码不存在 |
| 40091 | 邀请码已过期 |

---

### 4.2 通过邀请码加入企业

**POST** `/company/join`

**说明：** 用户通过邀请码加入企业，加入后 `company_id` 自动绑定到当前用户。

**请求体：**

```json
{
```


**错误码：**

| code | 说明 |

## 5. 摄像头管理 (Camera)

> **架构说明：** 摄像头通过 RTSP 协议直连后端服务，后端负责 RTSP → HLS 转码供前端播放，同时按巡检计划定时从 RTSP 流中抓帧送入 Python 推理服务。前端不直接与摄像头通信，所有视频流和抓拍操作均通过后端中转。
>
> ```
> 摄像头(RTSP) ──RTSP──> 后端(Java) ──HLS──> 前端(浏览器)
>                        │
>                        ├── 定时抓帧 → Python推理服务(8000)
>                        ├── 手动抓拍 → 存储 + 推理
>                        └── WebSocket推送 → 前端实时通知
> ```
>
> | 环节 | 协议 | 说明 |
> |------|------|------|
> | 摄像头 → 后端 | RTSP | IP 摄像头事实标准，内网直连，支持主码流/子码流 |
> | 后端 → 前端（直播） | HLS (.m3u8) | 浏览器原生支持，延迟 3-10s |
> | 后端 → 推理服务 | HTTP (URL/base64) | 抓帧后以 URL 或 base64 形式提交推理 |
> | 后端 → 前端（通知） | WebSocket | 推理结果、状态变更实时推送 |

### 5.1 摄像头列表

**GET** `/camera/list`

**请求参数：**

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| status | string | 否 | — | 状态筛选（ONLINE/OFFLINE/FAULT） |
| keyword | string | 否 | — | 搜索关键词（名称/RTSP地址） |
| page | int | 否 | 1 | 页码 |
| size | int | 否 | 20 | 每页条数 |

**响应：**

```json
{
  "code": 200,
  "data": {
    "records": [
      {
        "id": "uuid",
        "name": "A区-1号摄像头",
        "rtspUrl": "rtsp://192.168.1.100:554/stream1",
        "rtspUrlSub": "rtsp://192.168.1.100:554/stream2",
        "locationX": 120.5,
        "locationY": 35.2,
        "direction": 90.0,
        "coverageGrids": ["grid-uuid-1", "grid-uuid-2"],
        "captureResolution": "640x640",
        "captureQuality": 85,
        "reconnectInterval": 30,
        "status": "ONLINE",
        "lastOnlineAt": "2026-06-09T10:30:00",
        "createdAt": "2026-06-01T10:00:00",
        "updatedAt": "2026-06-09T10:30:00"
      }
    ],
    "total": 8,
    "page": 1,
    "size": 20
  }
}
```

**响应字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| rtspUrl | string | RTSP 主码流地址，用于抓拍和转码 |
| rtspUrlSub | string\|null | RTSP 子码流地址（低分辨率），用于前端实时预览时可降低带宽；为 null 时预览复用主码流 |
| captureResolution | string | 抓拍分辨率，格式 `宽x高`，默认 `640x640` |
| captureQuality | int | 抓拍 JPEG 质量 1-100，默认 85 |
| reconnectInterval | int | 断流重连间隔（秒），默认 30 |
| status | string | ONLINE / OFFLINE / FAULT |
| lastOnlineAt | datetime\|null | 最近一次在线时间 |

---

### 5.2 新增摄像头

**POST** `/camera`

**权限：** ADMIN

**请求体：**

```json
{
  "name": "A区-1号摄像头",
  "rtspUrl": "rtsp://192.168.1.100:554/stream1",
  "rtspUrlSub": "rtsp://192.168.1.100:554/stream2",
  "locationX": 120.5,
  "locationY": 35.2,
  "direction": 90.0,
  "coverageGrids": ["grid-uuid-1", "grid-uuid-2"],
  "captureResolution": "640x640",
  "captureQuality": 85,
  "reconnectInterval": 30
}
```

**请求字段说明：**

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| name | string | 是 | — | 摄像头名称，不可与已有名称重复 |
| rtspUrl | string | 是 | — | RTSP 主码流地址，格式 `rtsp://host:port/path` |
| rtspUrlSub | string | 否 | null | RTSP 子码流地址，用于低带宽预览 |
| locationX | float | 是 | — | 安装位置经度 |
| locationY | float | 是 | — | 安装位置纬度 |
| direction | float | 否 | 0 | 朝向角度 0-360 |
| coverageGrids | string[] | 否 | [] | 覆盖的网格ID列表 |
| captureResolution | string | 否 | "640x640" | 抓拍分辨率 |
| captureQuality | int | 否 | 85 | 抓拍 JPEG 质量 1-100 |
| reconnectInterval | int | 否 | 30 | 断流重连间隔（秒），范围 10-300 |

**校验规则：**
- `rtspUrl` 必须以 `rtsp://` 开头
- `captureQuality` 范围 1-100
- `reconnectInterval` 范围 10-300

**响应：**

```json
{
  "code": 200,
  "data": {
    "id": "uuid"
  },
  "message": "摄像头添加成功，正在尝试建立RTSP连接"
}
```

> **说明：** 新增摄像头后，后端立即尝试建立 RTSP 连接，连接成功后状态自动更新为 ONLINE，失败则为 OFFLINE。

**错误码：**

| code | 说明 |
|------|------|
| 40080 | RTSP 地址格式不正确 |
| 40081 | 摄像头名称已存在 |
| 40083 | RTSP 流连接失败（地址不可达或认证失败） |

---

### 5.3 修改摄像头

**PUT** `/camera/{id}`

**权限：** ADMIN

**请求体：** 同新增，所有字段均可部分更新

**校验规则：**
- 修改 `rtspUrl` 后，后端会断开旧连接并重新建立 RTSP 连接
- 修改 `rtspUrlSub` 不会影响主码流抓拍

---

### 5.4 删除摄像头

**DELETE** `/camera/{id}`

**权限：** ADMIN

**响应：**

```json
{
  "code": 200,
  "message": "摄像头删除成功，已断开RTSP连接并清理关联流资源"
}
```

> **说明：** 删除摄像头时，后端会断开 RTSP 连接、清理 HLS 转码缓存、停止该摄像头的所有巡检任务关联。已存储的抓拍图片和历史巡检日志不会删除。

```json

    "status": "ONLINE",

    "streamUrl": "/stream/camera-uuid.m3u8",
    "connectionInfo": {
      "protocol": "RTSP",
      "transport": "TCP",
      "uptime": 86400,
      "retryCount": 0
    },
    "frameInfo": {
      "width": 640,
      "height": 640,
      "fps": 25,
      "codec": "H.264"
    }
  }
}
```

**响应字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| status | string | ONLINE / OFFLINE / FAULT |
| lastFrameAt | datetime\|null | 最近一次收到帧的时间 |
| streamUrl | string\|null | HLS 播放地址，摄像头在线时返回 |
| connectionInfo.protocol | string | 连接协议，固定为 RTSP |

| connectionInfo.retryCount | int | 累计重连次数 |

**摄像头状态判定逻辑：**

| 状态 | 条件 |

| OFFLINE | RTSP 连接断开或最近 60s 内无帧到达 |

### 5.7 HLS 视频流

```
camera-uuid-0.ts
```

> **前端集成说明：** 前端使用 HLS.js 或 video.js 播放器，直接加载 m3u8 地址即可。HLS 延迟约 3-10s，满足监测场景需求。

### 10.6 推理服务 — 图片检测（Python 推理服务）

> **说明：** 以下接口由独立的 Python FastAPI 推理服务提供（默认端口 8000），负责加载 YOLO 模型执行病害 + 虫害双模型推理。Spring Boot 后端通过 HTTP 调用该服务。

**服务地址：** `http://localhost:8000`

**响应格式：** 与 Java 后端保持一致的 `{code, message, data}` 结构。

**模型信息：**

| 模型 | 类型 | 类别数 | 说明 |
|------|------|--------|------|
| disease | YOLOv8 | 38 | 植物病害检测（基于 PlantVillage 数据集，含苹果、番茄、玉米等 14 种作物） |
| pest | YOLOv8 | 102 | 农作物虫害检测（含稻纵卷叶螟、蚜虫、红蜘蛛等） |

每次推理对同一张图片**顺序执行**病害和虫害两个模型，合并结果后返回。

#### 10.6.4 标注图说明

推理服务同时执行病害和虫害检测后，将两类检测框绘制在**同一张图**上，用颜色区分：

| 类别 | 框颜色 | 标签前缀 | 示例 |
|------|--------|----------|------|
| 病害 | 红色 | `[病]` | `[病]番茄晚疫病 0.93` |

#### 10.6.5 并发控制说明

| 项目 | 说明 |
|------|------|
| 并发策略 | asyncio.Lock 串行化推理，同一时刻仅一个请求在执行模型推理 |
| 排队机制 | 后续请求在锁上等待，按到达顺序（近似 FIFO）依次执行 |
| 批量单位 | 批量请求中每张图片独立排队，其他请求可插入 |
| 阻塞保护 | 推理通过 `run_in_executor` 在线程池中执行，不阻塞 FastAPI 事件循环 |

> **性能参考（RTX 4060 Laptop 8GB）：** 首次推理含 GPU 预热约 1200ms，后续单张推理约 70-150ms（病害 + 虫害合计）。
