# 五、风险管控

1. 依赖风险：部分任务存在依赖关系，前置任务负责人需优先保障进度，每日站会同步依赖任务状态，避免后续任务阻塞；若依赖条件未达成，由敏捷教练协调资源解决。

2. 需求理解偏差风险：任务执行前，团队成员需与产品负责人确认需求细节，关键功能可通过原型评审、需求宣讲等方式统一认知，减少返工。

3. 进度滞后风险：按任务复杂度拆分工作包，设定关键时间节点，每日跟踪任务完成情况；若出现进度偏差，及时分析原因，通过增加协作支持、调整任务优先级等方式弥补。

4. 质量风险：开发过程中遵循编码规范，提交代码前进行本地单元测试，及时发现并跟踪缺陷修复，确保交付功能符合验收标准。

5. 沟通风险：建立即时沟通渠道（QQ群），重要信息通过会议纪要、文档同步等方式留存，确保团队成员及相关方信息一致。

6. 技术风险：推理服务部署不稳定时，准备Mock推理接口作为降级方案；前后端接口格式不一致时，严格按照接口文档V1.2开发。
