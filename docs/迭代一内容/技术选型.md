# **1.** ***\*前言\****

## **1.1.** ***\*编写目的\****

本文档旨在明确“农作物疾病监测系统”全生命周期的技术栈、算法中间件、人工智能（AI）协同方案及工程化工具链。通过引入行业先进的边缘深度学习推理架构与大语言模型（LLM）Agent 编排与 RAG 架构，保障系统开发的高效性与前沿性，为系统的软件著作权申请、技术面试复盘提供严谨的工程依据。

本文档旨在明确“农作物疾病监测系统”全生命周期的技术栈、算法中间件、数据协同体系及人工智能（AI）辅助方案。通过统一前沿技术选型标准，保障系统开发的一致性、高效性与稳定性，重点突出分布式业务架构与大模型（LLM）Agent、YOLOv8 边缘推理的工程化融合，为系统研发、测试、部署及软件著作权申请提供核心技术依据。

## **1.2.** ***\*适用范围\****

适用于“农作物疾病监测系统”项目全生命周期技术实施场景，包含无密码邮件安全认证、异构病害/虫害双线视觉推理、2.5D数字孪生时空看板、大模型日度农情诊断决策等核心模块的研发与部署。

适用于“农作物疾病监测系统”项目全生命周期的技术实施场景（含需求分析、架构设计、编码开发、算法打包、测试验证、部署运维阶段），覆盖研发、算法、测试、运维团队，作为技术落地与跨团队协同的最高执行依据；不含手机端相关技术选型与适配场景。

# **2.** ***\*技术选型原则\****

***\*一致性\****：统一前后端及 AI 推理微服务框架的版本，保障开发、测试、生产环境的兼容性；

***\*先进性\****：全面拥抱行业主流的 Vue 3 + Spring Boot 3 技术栈，摒弃过时组件，确保系统架构的前沿性；

***\*智能化\****：深度集成 LangChain4j 与边缘端 ONNX Runtime，提升大模型动态编排与多模态感知的高效性；

***\*解耦性\****：实行核心业务逻辑与重型深度学习推理的分离，通过轻量级异步协议互联，保障主骨干网的流畅性；

***\*闭环性\****：利用事件驱动机制与跨系统邮件网关回调，实现突发灾情工单从“感知到响应”的无人值守自动化闭环性；

***\*隔离性\****：引入基于 JWT 载荷的 RBAC 拦截机制，实现管理员与普通用户大棚数据的多租户安全隔离性；

***\*防御性\****：核心接口挂载基于 Redis 滑动窗口的流控限流器，保障无密码登录注册体系面对恶意爆破时的顽健性。

***\*先进性与高亮技能（Skills）结合\****：拒绝过时技术，前端全面拥抱 Vue 3 + Vite 生态，后端采用 Spring Boot 3.x + JDK 17，深度结合 GitHub 热门的微服务异步解耦方案；

***\*感知、认知与业务三层解耦\****：

***\*感知层\****：Python FastAPI + ONNX Runtime 处理多模型高并发视觉推理。

***\*认知决策层\****：Spring Boot 3.x + LangChain4j 编排大模型 RAG 上下文与 SSE 流式推送。

***\*业务层\****：Spring Security 6.x 维护无状态角色权限流转与安全 OTP。

***\*事件驱动与自动化闭环\****：系统的演进由“数据异样事件”驱动，通过工单状态机与跨系统邮件网关（Token 回调）实现工业级无人值守闭环。

## **4.2.** ***\*注意事项\****

l 版本兼容性：各 AI 工具生成的代码片段，必须严格经过本地编译器验证，确保完全兼容 Spring Boot 3.3.x、JDK 17 以及 Vue 3 的 Composition API 语法规范，避免引入陈旧或弃用的依赖 API；

l 提示词隔离性：在使用公有云大模型服务时，禁止将系统敏感的无密码登录验证凭证签名（JWT Secret）、企业物理设备真实 IP 映射、MySQL 核心密码等机密数据直接录入 Prompt，必须通过本地环境变量或配置文件进行安全隔离性；

l 人工审校性：由于大模型存在固有的“幻觉”特征，AI 代理（如 Claude Code）自主修改的代码必须通过本地单元测试验证，核心业务状态机（如工单流转逻辑、鉴权拦截器）必须由人工进行全量 Code Review，确保系统逻辑的绝对鲁棒性；

l 版权合规性：明确所选用的 AI 工具链与底层大模型均支持学术开源及商业级代码生成服务，确保生成的算法代码、架构脚手架符合开源合规性规范。

# **5.** ***\*技术风险与应对措施\****

| **风险点**                                     | **影响程度** | **应对措施**                                                 |
| ---------------------------------------------- | ------------ | ------------------------------------------------------------ |
| 双 YOLOv8 并发推理导致显存崩溃                 | 高           | 采用 RabbitMQ 作为流量缓冲区。Java 端将抓拍图片事件投递进队列，Python 推理端通过队列消费者机制进行有序消费，利用 ONNX Runtime 进行底层硬件级加速和多线程推理优化。 |
| 大模型 API 响应超时导致定时日报断链            | 中           | 利用 LangChain4j 的重试拦截器机制配置 3 次自动重试。若最终依然超时，系统自动激活本地降级策略——改由传统代码基于预设结构化模板进行代码级拼接，确保日报不卡死。 |
