## 二、风险等级定义

| 风险等级 | 判定标准 | 颜色标识 |
|----------|----------|----------|
| **高** | 发生概率高（≥60%）且影响程度严重，可能导致项目延期或核心功能不可用 | 🔴 |
| **中** | 发生概率中等（30%-60%）或影响程度较大，可能影响部分功能交付或进度 | 🟡 |
| **低** | 发生概率低（<30%）且影响程度可控，对项目整体影响较小 | 🟢 |

## 四、风险监控与应对机制

### 4.1 风险监控频率

| 风险等级 | 监控频率 | 监控方式 |
|----------|----------|----------|
| 🔴 高风险 | 每日监控 | 每日站会专项汇报，SM跟踪闭环 |
| 🟡 中风险 | 每迭代监控 | 迭代规划会议识别，迭代回顾会议复盘 |
| 🟢 低风险 | 定期监控 | 周度检查，异常时及时上报 |

### 4.2 风险应对策略

| 应对策略 | 适用场景 | 说明 |
|----------|----------|------|
| **规避** | 风险发生概率高且影响严重 | 调整计划或方案，避免风险触发 |
| **转移** | 风险可通过外部资源分担 | 如使用成熟开源框架降低技术风险 |
| **缓解** | 风险可采取措施降低概率或影响 | 如预留缓冲时间、准备降级方案 |
| **接受** | 风险影响可控且应对成本高 | 记录风险，做好应急预案 |

### 4.3 风险上报流程

```
发现风险 → 站会/即时沟通上报 → SM协调评估 → 制定应对措施 → 跟踪闭环
```

1. **发现风险**：团队成员在开发过程中识别潜在风险
2. **及时上报**：在每日站会或QQ群即时沟通中上报
3. **评估定级**：SM协调团队评估风险概率与影响
4. **制定措施**：明确责任人与应对措施
5. **跟踪闭环**：记录在风险跟踪表，持续监控直至闭环
