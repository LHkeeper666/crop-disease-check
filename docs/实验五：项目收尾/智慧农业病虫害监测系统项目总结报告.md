# 一、项目概述

## （一）项目基本信息

- **项目名称**：智慧农业病虫害监测系统
- **项目周期**：2026年6月8日 - 2026年6月18日
- **项目目标**：开发一套面向现代农业的智能化病虫害监测平台，整合物联网摄像头、YOLOv8 AI图像识别、大语言模型等技术，实现病虫害的自动识别、智能预警、工单管理和防治决策支持。
- **项目范围**：覆盖用户认证、图像上报、病虫害识别、智能工单、农情报表、实时监控、农业大脑、设备管理、系统管理等核心模块；排除移动端适配、第三方系统集成等非核心功能。

## （二）项目执行概况

项目采用敏捷Scrum框架，以迭代冲刺模式分三个阶段推进：迭代一（MVP核心流程，3天）、迭代二（功能完善与缺陷修复，2天）、迭代三（用户体验优化与创新功能，2天）。项目团队严格遵循敏捷开发流程，完成了需求分析、系统设计、开发实施、测试验收全流程工作，于2026年6月18日如期竣工，各项功能达到预设标准，系统运行稳定。项目累计提交318次Git记录，6名核心成员参与开发，覆盖后端、前端、算法、测试、文档等全职能角色。

# 二、项目完成情况

## （一）核心目标达成

已实现"摄像头抓拍→AI病虫害识别→置信度判定→智能工单生成→专家处理→数据统计→日报生成"全流程业务闭环，系统功能满足农业园区病虫害无人值守监测的实际应用需求。核心业务指标方面，识别响应时间≤3秒/张，达到预期目标；病虫害识别类别覆盖140类（38种病害+102种害虫），超额完成≥100类的目标；多租户企业级数据隔离架构、实时监控与WebSocket推送、SSE流式AI对话助手等功能均达成或超额达成预设标准。

## （二）交付物完成情况

按计划完成全部交付物输出，包括规划类文档（团队章程、开发计划表、用户故事文档、软件需求规格说明书、风险识别和管控表）、设计类文档（技术选型文档、数据库设计文档、接口文档）、迭代过程文档（三轮迭代的冲刺计划会议纪要、每日站会记录、用户操作手册）、开发类交付物（Spring Boot后端代码、Vue 3前端代码、Python CV推理服务代码、数据库初始化脚本、Docker Compose部署配置）以及测试类文档（各迭代测试报告、423个单元测试用例），所有资料已按规范归档，可满足后续查阅与运维需求。

## （三）项目里程碑达成

顺利完成项目启动、三次迭代上线、项目竣工五大里程碑节点。项目启动（2026.06.09）完成需求分析与团队分工；迭代一上线（2026.06.12）实现登录、图片上传、病虫害识别核心流程；迭代二上线（2026.06.16）完成审核-防治-统计流程闭环与Agent技能配置；迭代三上线（2026.06.18）实现角色权限精细化、戴夫AI宠物助手、实时检测优化等功能；项目竣工（2026.06.18）达成系统全流程闭环、缺陷清零与文档归档。各节点核心标志均如期实现，无重大延误情况。

# 三、成功经验

## （一）项目管理模式高效适配

采用Scrum敏捷框架，通过每日站会同步进度、快速解决问题，迭代回顾及时优化流程，确保项目推进方向准确。明确各迭代阶段核心目标与交付标准，迭代一聚焦MVP核心流程，迭代二聚焦缺陷修复与功能完善，迭代三聚焦用户体验与创新功能，层层递进。任务拆分合理，进度管控精细化，有效保障了项目按期交付。

## （二）团队协作机制健全

建立清晰的角色分工与职责体系，产品负责人（黄博）主导需求管理与决策担当，敏捷教练（陈嘉豪）负责流程保障与障碍清除，后端开发（陆华飞）承担核心业务逻辑，算法开发（邵建疆、粟昱斌）负责YOLOv8模型与推理服务，前端开发（陈思宇）负责页面与文档管理，各角色各司其职、协同配合。跨角色沟通渠道畅通（QQ群即时沟通、Git代码托管、U+平台文档协作），关键节点组织专项对接会议，确保需求传递、技术衔接无偏差，提升了整体工作效率。

## （三）技术选型科学合理

结合项目需求与实际应用场景，选定Spring Boot 3 + JDK 17 + MyBatis-Plus、Vue 3 + Vite + Element Plus、Python FastAPI + YOLOv8等技术栈，兼顾了开发效率、系统稳定性与扩展性。引入LangChain4j编排大模型实现SSE流式对话与工具调用，RabbitMQ异步消息实现事件驱动工单闭环，MinIO对象存储承载大文件资产，ElasticSearch支撑全文检索。采用Docker Compose Profile分离中间件与应用服务，保障了环境一致性，减少了部署冲突，提升了上线效率。

## （四）AI辅助编程深度应用

项目全程深度使用Claude Code等AI辅助工具，显著提升了开发效率。AI工具在全栈代码重构、自动化测试、Debug调试、文档生成等环节发挥了重要作用，帮助团队在有限时间内完成了140类病虫害识别、SSE流式对话、2.5D热力图数字孪生看板、戴夫AI桌面宠物等复杂功能的开发。

## （五）风险管控提前落地

项目初期建立完善的风险识别与应对表，针对需求、技术、进度、团队、环境、质量、安全七大类24项风险制定专项应对措施；过程中持续跟踪风险状态，提前预判潜在问题并及时处置，有效规避了重大风险对项目的影响。进度风险（高风险）通过每日站会、任务拆分、缓冲时间等措施得到有效控制。

# 四、存在的问题与教训

## （一）需求管理精细化不足

需求分析阶段对部分跨模块流程细节挖掘不够深入，核心逻辑确认缺乏书面化闭环机制，如检测阈值与工单联动逻辑、农情报表数据源一致性等问题，在开发过程中才暴露出来，引发局部返工，影响了部分节点进度。教训：需求管理需强化全流程规范化，关键逻辑与验收标准必须形成书面确认文件，确保所有相关方认知一致，跨模块依赖关系需在迭代规划阶段充分识别。

## （二）迭代周期规划偏紧

原计划7天完成3轮迭代，实际执行延长至11天。迭代一遗留6项功能延至迭代二，迭代二遗留5项功能延至迭代三，迭代三仍有3项功能未完成。任务估算偏乐观，缓冲时间不足。教训：任务估算需更加保守，预留充足的缓冲时间，P0级功能应集中在前两轮迭代完成，第三轮迭代以优化和收尾为主，故事点估算需考虑前后端联调、跨模块依赖等隐性成本。

## （三）前后端联调效率待提升

部分前后端接口设计与联调阶段，因前期沟通不充分、接口规范交底不到位，出现联调受阻情况，如工单ID类型不兼容、Controller路径与context-path重复导致404、JWT拦截器路径配置等问题，消耗了较多调试时间。教训：跨模块、跨角色协作需建立前置交底机制，接口设计完成后需组织专项评审，前后端并行开发时Mock数据先行可有效减少联调阻塞。

## （四）Git分支管理规范执行不严格

项目过程中存在少量未严格遵循Git分支管理规范的情况，部分提交信息不够规范（如"update"、"1"等无意义信息），代码合并冲突处理不够及时，主分支直接提交的情况较多，缺乏Code Review机制。教训：需强化Git分支管理规范的执行与监督，建立PR + Code Review机制，确保代码质量，提交信息需遵循约定式提交规范。

## （五）技术文档同步更新滞后

部分技术文档的更新滞后于代码开发，如接口文档与实际接口存在不一致、数据库设计文档未及时同步新增字段等，文档编写集中在迭代末期，存在赶工现象。教训：技术文档应与代码开发同步更新，建立文档即代码的理念，接口变更时同步更新Knife4j文档，数据库变更时同步更新设计文档。

# 五、后续改进建议

## （一）优化需求管理流程

修订需求管理规范，明确需求调研、评审、确认、变更的全流程标准，要求关键需求与流程逻辑必须经多方签字确认；建立需求答疑与评审机制，确保团队全员对需求理解一致，跨模块依赖关系需在迭代规划阶段充分识别并文档化。

## （二）完善迭代计划编制

任务估算需更加保守，引入三点估算法（乐观/最可能/悲观）提升估算准确性；预留充足的缓冲时间应对突发情况；P0级功能应集中在前两轮迭代完成，第三轮迭代以优化和收尾为主；建立迭代容量模型，避免任务过载。

## （三）强化前后端协作机制

建立前后端接口前置评审制度，接口设计完成后组织专项评审确认；前后端并行开发时优先完成接口规范文档，Mock数据先行；使用Knife4j等工具自动生成接口文档，减少手动维护成本；建立前后端联调检查清单，确保接口一致性。

## （四）严格Git分支管理规范

建立PR + Code Review机制，所有代码合并需经过至少1人审查；提交信息遵循约定式提交规范（feat/fix/docs/refactor等前缀）；建立分支保护规则，主分支禁止直接提交；定期进行代码质量扫描，确保代码规范一致性。

## （五）沉淀项目经验资产

整理项目全流程经验与教训，形成标准化开发流程模板与风险应对手册，沉淀技术最佳实践（Docker部署配置、LangChain4j集成方案、YOLOv8推理服务架构等），应用于后续同类项目，提升项目开发效率与质量。
